STM32F411 智能家居环境监测系统

STM32F411 系统实物运行 · DHT11 / DS18B20 传感器采集与 TFT 显示

📌 项目背景与动机

在学习 STM32 外设驱动的过程中，我先后完成了 GPIO、SPI、UART、DMA、ADC、TIM 等模块的单独验证。但这些独立实验无法体现一个完整嵌入式系统的复杂性——真正的挑战在于如何让多个外设在同一时间轴上协同工作，同时保证界面流畅、传感器采集可靠、通信稳定。

因此，我决定以智能家居环境监测终端为应用场景，将所学外设知识整合成一个从硬件设计到软件架构的完整系统。这个项目的核心目标不是"把模块跑通"，而是在裸机 HAL 环境下实现一个响应及时、架构清晰、便于扩展的嵌入式终端。

🏗️ 系统架构

传感器层 DHT11 DS18B20 ADC ×3 → STM32F411CEU6
10ms 分时调度器 → TFT 菜单显示
(SPI 50MHz DMA) 蜂鸣器报警 → 按键菜单
(事件队列) 通信层 ESP8266
UART AT · 状态机 → MQTT 云端上传 → UART 日志
FIFO 异步输出

核心设计思想：系统基于 10ms 软件节拍（HAL_GetTick 基准），通过分时调度器驱动 9 个独立任务。TFT 刷新保持 50Hz，传感器采集不阻塞主循环，ESP8266 通信走状态机链式推进，UART 日志经 FIFO 缓冲后中断发送。主循环中零 HAL_Delay、零 while 忙等。

🔧 关键难点与解决方案

难点一：SPI + DMA 驱动 ST7789V TFT — 从阻塞闪烁到 50fps 流畅刷新

❖ 遇到的问题

ST7789V 彩屏分辨率 320×240（RGB 565 格式），单帧像素数据量约 150KB。最初使用阻塞式 SPI 逐帧发送时：

  • 全屏刷新卡顿严重：CPU 被大量像素传输完全占用，主循环其他任务被挂起，用户体验差。
  • 界面闪烁：全屏逐行刷新的过程中，新旧帧交替可见，视觉上产生撕裂感。
  • 与其他任务互斥：传感器采集、按键扫描必须在刷屏间隙执行，时序难以协调。

❖ 我的解决思路

核心矛盾是"大量数据传输占用 CPU"与"系统需要响应其他任务"。我沿着两个方向破局：

  1. 传输层面：将像素数据搬运从 CPU 卸载到 DMA，CPU 仅负责启动传输。
  2. 策略层面：不必每次全屏重绘，只刷新发生变化的"脏区域"。

❖ 具体实现

  • SPI 50MHz + DMA 异步传输：将 ST7789V 的 SPI 速率推至 50MHz，启动 DMA 传输后 CPU 立即返回。通过全局标志 tft_spi_dma_busy 判断当前 DMA 是否空闲——若忙则跳过本帧，避免数据冲突。
  • DMA 完成回调：在 HAL_SPI_TxCpltCallback 中清除 busy 标志，通知调度器可发送下一帧。
  • 三级刷新策略：
    • 无变化时：TFT 任务直接返回，零开销。
    • 数据变化时：仅局部刷新数值区域（~2ms），用户感知不到延迟。
    • 页面切换时：全屏重绘（~24ms），仅发生在按键触发时，频率极低。
  • 调度器 20ms 周期：TFT 任务每 20ms 执行一次，对应 50Hz 刷新频率，与 DMA 异步机制配合，CPU 利用率大幅下降。

❖ 达到的效果

  • DMA 优化后 清屏时间从数百毫秒降至 < 1ms。
  • 界面刷新率达到 50+ fps，菜单切换和传感器数值更新流畅无闪烁。
  • 传感器采集和 MQTT 上传不再被 TFT 刷新阻塞，多任务协同运行。
void TFT_Update_Task(void) { if (tft_spi_dma_busy) return; // DMA 忙 → 跳过本帧，不阻塞 if (ui_need_full_redraw) { // 页面切换 → 全屏重绘 ~24ms TFT_Show_UI_Full(); ui_need_full_redraw = 0; } else if (ui_data_dirty) { // 数据变化 → 局部刷新 ~2ms TFT_Update_Data_Area(); } // 无变化 → 直接返回，零 CPU 开销 }

难点二：DS18B20 非阻塞状态机 — 消灭主循环中的 750ms 硬等待

❖ 遇到的问题

DS18B20 温度传感器的温度转换时间典型值为 750ms，这是硬件物理极限，无法通过软件加速。原始驱动中调用 HAL_Delay(750) 等待转换完成，这意味着：

  • 主循环每采集一次温度，就有 750ms 完全卡死——TFT 动画停滞、按键无响应、MQTT 断连。
  • 如果同时采集 DHT11（~20ms 阻塞）并上传 MQTT（AT 指令串行等待 ~12s），整个系统几乎不可用。

❖ 我的解决思路

"等待"本身不是问题——问题是"谁在等"。CPU 不应该亲自等 750ms，应该启动转换后立即去做其他事，到时间了再回来取结果。这正是非阻塞状态机的典型应用场景。

❖ 具体实现

  • 四阶段状态机：将采集流程拆分为 IDLE → START_CONVERT → WAIT_CONVERT（750ms）→ READ_SCRATCHPAD → PROCESS_DATA → IDLE。
  • 基于 10ms 节拍的非阻塞计时：启动转换时记录当前 g_tick_10ms，后续每次任务被执行时检查 tick 差值是否 ≥ 75（即 750ms），取代 HAL_Delay。
  • 1 秒调度周期：DS18B20 任务每 100 tick（1s）执行一次，完整采集周期约 2 秒。这个频率对于环境温度监测完全足够，且不给系统增加负担。
  • 容错处理：读取失败时保留上次有效值，防止 TFT 显示跳变或出现异常数值。

❖ 达到的效果

  • DS18B20 采集不再阻塞主循环，750ms 等待期间 CPU 正常执行 TFT 刷新、按键扫描、MQTT 上传等任务。
  • 温度数据每 ~2s 更新一次，满足环境监测的实时性需求。
  • 验证方法：按下按键时菜单即时响应，不会出现"按键没反应、要等传感器读完才动"的情况。
void DS18B20_Task(void) { switch (ds18b20_state) { case DS18B20_STATE_IDLE: DS18B20_StartConvert(); // 发送转换命令 → 立即返回 ds18b20_convert_start_tick = g_tick_10ms; // 记录启动时间 ds18b20_state = DS18B20_STATE_WAIT_CONVERT; break; case DS18B20_STATE_WAIT_CONVERT: if (g_tick_10ms - ds18b20_convert_start_tick >= 75) { // 750ms 到了？ DS18B20_ReadScratchpad(&lsb, &msb); // 读取结果 → 立即返回 ds18b20_state = DS18B20_STATE_PROCESS_DATA; } // 时间未到 → 直接返回，CPU 处理其他任务 break; case DS18B20_STATE_PROCESS_DATA: g_data.ds18b20_temp = (lsb + (msb << 8)) * 0.0625f; g_data.ds18b20_valid = 1; ds18b20_state = DS18B20_STATE_IDLE; // 回到起点，下一轮采集 break; } }

难点三：ESP8266 + MQTT — 从串行阻塞 12 秒到状态机链式推进

❖ 遇到的问题

ESP8266 通过 UART AT 指令连接 Wi-Fi 并接入 MQTT，标准流程需要串行发送 9 条 AT 指令（AT 测试 → 回显关闭 → 设置模式 → 连接 Wi-Fi → 连接 MQTT → 发布数据）。原始实现中每条指令后 HAL_Delay(500~5000ms) 等待响应，总计阻塞约 12 秒，且多个传感器数据连续上传时串口缓冲区压力大，出现丢包。

❖ 解决思路与实现

  • AT 指令状态机：将 9 条指令拆分为状态链 POWER_ON → AT_TEST → ECHO_OFF → SET_MODE → CONNECT_WIFI → CONNECT_MQTT → MQTT_READY → MQTT_PUBLISH → IDLE，每 10ms 推进一个步骤，用 g_tick_10ms 做超时判断替代 HAL_Delay。
  • MQTT 分时上传：温度、湿度、ADC 等数据分散到不同时间片发布（每 2s 一个 topic），降低瞬时串口通信压力。
  • MQTTX 验证：使用 MQTTX 客户端订阅对应 topic，验证云端消息到达率和数据准确性。

❖ 效果：ESP8266 初始化不再卡死主循环，AT 指令失败时状态机可超时重试，MQTT 数据稳定上传。

难点四：PCB 供电干扰 — 多模块同时工作时系统复位

❖ 遇到的问题

PCB 打样焊接后，ESP8266 联网、TFT 全屏刷新和蜂鸣器同时工作时，STM32 偶发复位。问题定位到模块启动瞬间的大电流导致 3.3V 轨电压跌落，触发 BOR（Brown-Out Reset）。

❖ 解决思路与实现

  • 硬件：每个 VDD 引脚就近放置 100nF 去耦电容 + 电源入口增加 10μF 钽电容；加宽 TFT 背光和 ESP8266 供电走线，降低线路压降。
  • 软件：分时启动策略——先初始化传感器和 TFT，延迟 500ms 后再给 ESP8266 上电，避免启动冲击叠加。

❖ 效果：系统长时间运行稳定，不再出现异常复位。

  • 串口日志：通过 UART FIFO 异步输出系统运行状态、ESP8266 AT 指令响应和 MQTT 上传结果，不阻塞主循环。

  • 按键测试：长短按识别、事件队列、页面切换全路径覆盖。

💡 项目复盘

这个项目最核心的收获是理解了嵌入式系统的本质不是外设驱动的堆叠，而是时序与调度的工程。TFT 刷新（20ms）、传感器采集（750ms~2s）、AT 通信（秒级）、按键扫描（10ms）各有完全不同的时间尺度——如果不用状态机 + 分时调度把这些任务的时间轴解耦，系统就会被最慢的任务拖垮。

通过这次重构，我建立了几个关键认知：

  • 裸机不等于简单：没有 RTOS 的情况下，通过软件节拍 + 状态机 + DMA 同样可以实现"看起来像是多任务并发"的效果，而且对调度逻辑有更彻底的控制。
  • 硬件和软件要一起调：PCB 能画出来 ≠ 系统能稳定跑。去耦电容的位置、电源走线的宽度、模块上电顺序——这些在原理图上看不到的东西，往往是最难排查的 bug 来源。
  • 可维护性从一开始就要考虑：CubeMX 可重新生成、调度周期常数集中管理、任务函数边界清晰——这些看似"多余"的设计，在后续迭代和调试中节省了大量时间。
