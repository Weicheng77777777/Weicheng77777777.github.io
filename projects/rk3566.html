BSP 调试流程

在这个项目中，我逐步建立了 BSP 调试的基本闭环：每次修改设备树或配置后，都需要重新编译对应的 DTB 或镜像，确认烧录到正确位置，再通过串口日志、dmesg、/proc/device-tree 和 /sys 节点验证修改是否真正生效。

关键实践

1. Kernel、DTB、rootfs 与启动镜像关系

在嵌入式 Linux 系统中，Kernel、DTB 和 rootfs 分别承担不同角色。Kernel 负责系统核心功能，DTB 描述板级硬件资源，rootfs 提供用户态运行环境。如果只修改设备树但没有重新编译或烧录正确的 DTB，板端运行结果不会发生变化。

通过本项目，我理解了每次修改设备树后需要重新编译 DTB，并确认镜像烧录或替换到正确位置。这个过程帮助我建立了 Linux BSP 开发中"源码修改"和"板端生效"之间的对应关系。

2. 设备树修改与验证

设备树是 Linux 内核识别板级硬件的重要依据。调试外设时，需要关注 compatible、reg、interrupt、pinctrl、status 等字段是否与硬件连接和驱动匹配。

我通过修改设备树节点并重新编译 DTB，在板端使用 /proc/device-tree 验证节点是否生效，并结合 dmesg 查看驱动 probe 情况。这个过程让我理解了设备树不是单纯配置文件，而是驱动匹配和硬件描述的关键入口。

3. 显示接口调试认知

RK3566 平台涉及 HDMI、eDP、MIPI 等多种显示接口，不同显示链路在设备树节点、供电、时钟、背光、panel 参数和接口连接上都有差异。

在学习过程中，我对比了不同显示接口的配置方式，理解了显示调试不能只看屏幕是否点亮，还需要检查设备树节点、panel 参数、背光控制、接口选择和内核日志。这个实践为后续做 LCD / MIPI / RGB 显示适配打下了基础。

4. GT9271 触摸模块 I2C 调试

GT9271 触摸模块通过 I2C 与主控通信，调试时需要确认 I2C 地址、设备树节点、interrupt 引脚、reset 引脚和驱动 compatible 是否匹配。

我围绕 I2C 设备识别、设备树节点配置和驱动 probe 日志进行排查，并通过 /sys/bus/i2c/devices、dmesg 和 input 设备相关信息验证触摸模块是否正常加载。这个过程让我理解了 I2C 触摸设备从设备树描述到驱动匹配再到 input 事件注册的基本路径。

项目复盘

这个项目让我建立了嵌入式 Linux BSP 开发的基本流程认知。和单片机项目不同，Linux BSP 调试不是改完代码就直接运行，而是需要经历配置修改、编译、烧录、上板启动、日志分析和再次修改的循环。尤其是设备树修改后，必须确认 DTB 重新编译并真正烧录到板端，否则修改不会生效。

通过显示接口和 GT9271 触摸模块调试，我理解了设备树在 Linux 驱动匹配中的作用。外设能否正常工作，不只取决于驱动是否存在，还取决于设备树中的 compatible、I2C 地址、中断引脚、reset 引脚和 status 等配置是否正确。

这个项目虽然是跟随官方文档完成的阶段性训练，但它帮助我建立了 Linux BSP 的入门框架，也为后续 GEC6818 ARM64 Linux 系统移植项目中的 DTS/DTB、LCD 和触摸适配打下了基础。
