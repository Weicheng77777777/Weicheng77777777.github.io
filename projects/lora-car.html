LoRa 遥控坦克小车系统

双 MCU · LoRa 无线通信 · GPS 定位 · 电机控制 · ADC 回传

▷ 查看 GitHub 仓库

项目背景

基于 STM32F4 和 STM32F1 设计双 MCU 遥控坦克小车系统。车端 STM32F4 负责运动控制、 传感器采集和数据回传，控制端 STM32F1 负责人机交互、指令发送和状态显示。 实现 LoRa 无线通信、GPS 定位解析、ADC 数据回传和电机驱动控制， 覆盖嵌入式系统中常见的双机通信、无线传输、传感器融合和电机控制等核心场景。

硬件组成

  • 车端 MCU：STM32F4 — 运动控制、数据采集
  • 控制端 MCU：STM32F1 — 遥控输入、人机交互
  • 无线通信：LoRa 模块 (UART 接口)
  • 定位：GPS 模块 (NMEA 协议解析)
  • 电机驱动：DRV8701 驱动直流减速电机
  • 显示：OLED 屏幕

关键难点与解决方案

1. LoRa 通信稳定性与数据包格式

LoRa 无线通信受距离和环境干扰影响，传输过程中可能出现丢包和乱序。 设计自定义数据包格式，加入帧头、帧尾和 CRC 校验，确保数据完整性。

/* 数据包格式 */ typedef struct { uint8_t header; // 帧头 0xAA uint8_t cmd; // 命令类型 uint8_t data[8]; // 数据负载 uint8_t checksum; // CRC 校验 uint8_t footer; // 帧尾 0x55 } lora_packet_t;

2. GPS NMEA 数据解析

GPS 模块输出 NMEA 0183 格式数据，需要解析 $GPGGA 和 $GPRMC 语句提取经纬度、 时间、海拔和定位状态等信息。在串口中断中逐字符解析，状态机管理解析流程。

3. 电机运动与通信的协同调度

电机动作、串口通信和 LoRa 数据传输需要协同工作，采用超级循环 + 状态机组织， 将遥控、回传和显示任务分配到不同时间片，避免任务饿死。

最终结果

✅ 双 MCU 系统通信正常

复盘总结

  • 双 MCU 系统的核心是 通信协议设计 — 帧格式、校验、超时重传缺一不可
  • LoRa 虽然传输距离远，但 速率和稳定性需要权衡
  • 状态机 是组织多任务嵌入式系统的核心思想
  • GPS NMEA 解析看似简单，但 异常字符和断帧处理 才是工程难点
