GEC6818 ARM64 Linux 系统移植

基于 S5P6818 Cortex-A53 · 板级 bring-up · DTS/DTB · LCD · Touch

GEC6818 开发板运行 ARM64 Linux · LCD RGB 800×480 正常显示

本项目基于 S5P6818 Cortex-A53 开发板，完成 GEC6818 从参考 BSP 到 ARM64 Linux 可运行系统的板级适配。 实现了 SD 卡启动 ARM64 Linux、GEC6818 专属 DTS/DTB、RGB LCD 显示、PWM 背光、FT5206 I2C 触摸和 I2C 外设验证。

我的主要工作

本项目基于 FriendlyElec Smart6818 ARM64 FriendlyCore 作为参考 BSP，我主要完成 GEC6818 开发板的板级差异适配：

  • 修正 U-Boot 阶段 DTB 加载路径，解决 Kernel panic
  • 新建 GEC6818 专属 DTS 文件，修改 model、compatible 和外设节点
  • 将显示链路从 LVDS / 1024×600 适配为 RGB / AT070TN92 / 800×480
  • 添加 LCD_EN GPIO、PWM 背光和 RGB pinctrl 配置
  • 修正 FT5206 触摸的 I2C bus、INT GPIO、WAKE GPIO 配置
  • 使用 dmesg、i2cdetect、/proc/device-tree、/proc/bus/input/devices 验证适配结果

系统架构

SD Card Boot → U-Boot → Load Kernel Image → Load GEC6818 DTB Linux Kernel 4.4.172 ARM64 → Device Tree Match → RGB LCD FT5206 Touch I2C Sensor

关键难点

1. DTB 加载错误导致 Kernel panic

首次启动时，Kernel 在早期阶段 panic。通过串口日志发现 Machine model 显示为 FriendlyElec NanoPi3，而不是 GEC6818。

我判断 Kernel image 已经被加载，问题更可能出在 DTB 加载路径或设备树内容。随后检查 U-Boot 启动参数和 DTB 文件，发现实际加载的是 NanoPi3 的 DTB。

// 串口日志关键输出 [ 0.000000] Machine model: FriendlyElec NanoPi3 [ 0.000000] Kernel panic - not syncing: Unable to ...

最终我新建 s5p6818-gec6818-rev01.dts，修改 model 和 compatible 字段，并修正 U-Boot 阶段 DTB 加载配置。

// 修复后验证 # cat /proc/device-tree/model GEC6818 based on S5P6818
# cat /proc/device-tree/compatible samsung,gec6818,samsung,s5p6818

2. LCD RGB 显示链路适配

参考系统使用 NanoPi3 的 LVDS / 1024×600 显示链路，而 GEC6818 使用 AT070TN92 RGB 接口 800×480 屏幕。

我将显示链路从 dp_drm_lvds 切换为 dp_drm_rgb，添加 AT070TN92 面板参数，恢复 RGB pinctrl，并增加 LCD_EN GPIO 与 PWM 背光控制。最终 LCD 屏幕正常显示，见上方实物图。

关键修改：
  • dp_drm_lvds → dp_drm_rgb
  • 在 panel-friendlyelec.c 中添加 AT070TN92 面板参数
  • 恢复 RGB pinctrl 配置
  • 添加 LCD_EN GPIO 和 PWM 背光控制

3. FT5206 触摸 probe 失败

原设备树中的触摸路径与 GEC6818 实际硬件不一致，导致 FT5206 probe 失败。 原路径为 i2c_2 + GPIOC16 + one-wire，而 GEC6818 实际使用 i2c_1 + GPIOB26(INT) + GPIOB8(WAKE) + FT5206。

调试命令链：
  • i2cdetect -y 1 — 确认 I2C 地址 0x38
  • dmesg | grep ft5206 — 查看驱动 probe 日志
  • cat /proc/bus/input/devices — 验证 input 设备注册
  • hexdump /dev/input/event1 — 验证触摸事件流
// hexdump 验证触摸事件流 # hexdump /dev/input/event1 0000000 00b4 0000 5f05 0005 0003 0000 01a8 0000 0000010 00b4 0000 5f05 0005 0003 0001 0106 0000

通过以上命令链逐步排查，确认 I2C 地址、中断引脚和驱动匹配关系，修复后触摸功能正常。

项目复盘

这个项目让我真正理解了设备树在嵌入式 Linux 中的作用。驱动能否正常 probe，不只取决于驱动代码是否存在，还取决于 compatible、reg、interrupt、pinctrl、gpio 等硬件描述是否与实际板卡一致。

同时，我也建立了比较完整的 bring-up 排查顺序：先确认 U-Boot 是否正确加载 Kernel 和 DTB，再看 Kernel early log，然后确认 machine model、driver probe、/proc 和 /sys 中的运行状态。

参考相近开发板的配置可以大幅减少试错成本，但必须验证每个差异点——看似相同的 SoC 在不同板卡上的外设引脚、I2C 地址、显示链路都可能不同。
