GEC6818 ARM64 Linux 系统移植

基于 S5P6818 Cortex-A53 · 板级 bring-up · DTS/DTB · LCD · Touch

▷ 查看 GitHub 仓库

项目背景

GEC6818 开发板基于 S5P6818 Cortex-A53 处理器，但官方资料以 32 位系统为主。 本项目尝试移植 ARM64 Linux，基于 FriendlyElec Smart6818 的 ARM64 FriendlyCore 系统， 在 GEC6818 开发板上跑通 SD 卡启动链路，并逐步完成 U-Boot、Kernel、DTS/DTB、LCD、触摸等板级适配工作。 项目使用 Git tag / GitHub Release 记录阶段性成果，将项目拆分为 v0.1.0~v0.3.0 等版本， 分别固化 DTS bring-up、LCD 适配和触摸适配结果。

  • 系统：FriendlyCore ARM64 (Linux 4.4.172)

关键难点与调试过程

1. DTB 加载失败导致 Kernel panic

首次启动时，U-Boot 加载 DTB 后 Kernel 直接 panic。 通过串口日志定位发现 U-Boot 阶段 DTB 地址配置错误， 实际加载的是 NanoPi3 的 DTB 而非 GEC6818 的 DTB。

// 串口日志关键输出 [ 0.000000] Machine model: FriendlyElec NanoPi3 [ 0.000000] Kernel panic - not syncing: Unable to ...

解决方案：修改设备树 model 和 compatible 字段， 创建 GEC6818 专属 DTS 文件 s5p6818-gec6818-rev01.dts， 并通过 cat /proc/device-tree/model 验证。

// 验证命令 # cat /proc/device-tree/model GEC6818 based on S5P6818
# cat /proc/device-tree/compatible samsung,gec6818,samsung,s5p6818

2. LCD 显示链路适配

原系统使用 NanoPi3 的 LVDS / 1024×600 链路，GEC6818 使用 RGB / AT070TN92 / 800×480。 需要切换驱动链路、添加面板描述、恢复 RGB pinctrl 并添加 LCD_EN 与 PWM 背光控制。

关键修改：
  • dp_drm_lvds → dp_drm_rgb
  • 在 panel-friendlyelec.c 中添加 AT070TN92 面板参数
  • 恢复 RGB pinctrl 配置
  • 添加 LCD_EN GPIO 和 PWM 背光控制

3. FT5206 触摸 probe 失败

旧设备树路径为 i2c_2 + GPIOC16 + one-wire，正确路径应为 i2c_1 + GPIOB26(INT) + GPIOB8(WAKE) + FT5206。 通过 /sys/bus/i2c/devices 和串口日志比对，发现 I2C 地址和中断引脚配置错误。

调试工具：
  • dmesg | grep ft5206 — 查看驱动 probe 日志
  • cat /proc/bus/input/devices — 验证输入设备注册
  • hexdump /dev/input/event1 — 验证触摸事件流
  • i2cdetect -y 1 — 扫描 I2C 总线设备
// 验证触摸事件流 # hexdump /dev/input/event1 0000000 00b4 0000 5f05 0005 0003 0000 01a8 0000 0000010 00b4 0000 5f05 0005 0003 0001 0106 0000

复盘总结

这次移植让我深刻理解了：

  • 设备树是 Linux 描述硬件的核心 — 节点和 compatible 必须与驱动匹配，否则外设无法 probe
  • 串口日志是最重要的调试手段 — 从第一行日志开始顺序分析，不要跳过任何 INFO/WARNING
  • 参考相近开发板的配置 — 可以大幅减少试错成本，但必须验证每个差异点
  • 版本管理对 BSP 开发至关重要 — Git tag 让每个阶段都可复现、可回滚
