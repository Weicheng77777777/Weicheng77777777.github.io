## 本阶段踩坑记录（2026-05-25）

### 坑 1：Edit 工具无法匹配含中文的字符串
**现象**：在修改 `gec6818.html` 时，Edit 工具反复报 `String to replace not found in file`，即使字符串内容完全正确。尝试过调整换行、调整空白字符、精简匹配范围，均失败。

**原因推测**：Edit 工具底层字符串匹配对 UTF-8 多字节字符（特别是中文）存在匹配缺陷。含中文的 `old_string` 无法正确匹配文件中的实际内容。

**解决方案**：放弃 Edit 工具处理含中文的 HTML 修改，改用 Bash + sed 进行行级操作：
- 先 `grep -n` 确定目标行号
- 再用 `sed -i '行号c\替换内容'` 进行替换
- 或用 `sed '行号a\内容'` 进行行后插入

### 坑 2：sed 插入了 tab 而非空格，导致缩进不一致
**现象**：使用 sed 插入 HTML 代码块后，新插入的代码使用 `\t`（制表符）缩进，而原文件全部使用空格缩进。在 cat -A 中表现为 `^I` 开头。

**解决方案**：用 `sed -i '行号范围s/^\t//'` 去掉前导 tab，或直接用 `sed -i 's/\t/      /g'` 全局替换。建议以后在 sed 插入内容时直接使用空格（每层 2 空格），不要用 `\t`。

### 坑 3：CRLF 与 LF 行尾混用
**现象**：文件是 Windows CRLF 行尾，在 Git Bash (Cygwin) 环境下操作时，某些工具表现不一致。PowerShell 的 `Get-Content -Raw` 保留 CRLF，但字符串比较时可能因不可见字符失败。

**解决方案**：统一在 Bash 环境中用 sed 操作，避免跨环境字符串匹配。如果需要用 PowerShell，用 `-Encoding utf8` 确保编码一致性。

### 推荐工作流（修改含中文的 HTML）
1. 用 Read 工具定位目标行号和周围内容
2. 用 `cat -A` 检查缩进字符（tab vs space）和行尾
3. 用 `sed -i '范围s/旧/新/'` 或 `sed -i '行号c\完整替换'` 执行修改
4. 用 `sed -n '范围p' | cat -A` 验证结果
5. 尽量避免 Edit 工具涉及多字节字符的替换操作

### 坑 4：Git 提交含 LF 行尾的文件时出现 CRLF 警告
**现象**：`git diff --stat` 时出现 `LF will be replaced by CRLF the next time Git touches it` 的警告。

**原因**：文件原为 CRLF 行尾（Windows），但用 Bash + sed 修改后，sed 写入的是 LF 行尾（Linux 风格）。Git 的 `core.autocrlf` 机制检测到不一致。

**解决方案**：
- 这个警告只是通知，不影响提交。Git 会在下次 checkout 时自动转换回 CRLF。
- 提交前先用 `git diff` 确认实际改动的 diff 是否正确，不要被警告干扰判断。
- 也可以配置 `git config core.autocrlf false` 关闭自动转换，但在 Windows 团队协作中不建议这样做。

### 坑 5：Git 需要单独处理二进制文件（PNG 图片）
**现象**：新增的 PNG 文件显示为 Untracked files，需要用 `git add` 单独添加。

**解决方案**：
- Git 会自动检测 PNG 为二进制文件，不会尝试做 diff 或行尾转换。
- 提交前确认 `git status` 中只包含了你想要的变更文件。
- 注意不要误提交不需要的文件夹（如 `.claude/`、大文件夹等）。

### 坑 6：一个文件同时处于 staged 和 unstaged 状态
**现象**：修改了已 staged 的文件后，`git status` 显示该文件既在 "Changes to be committed" 又出现在 "Changes not staged for commit" 中。

**原因**：`git add` 保存的是某个时刻的快照。之后再修改但没重新 `git add`，就会出现这个状态。

**解决方案**：
- 用 `git add <file>` 重新暂存最新版本
- 或者用 `git status` 确认当前状态后，确保所有修改都已 staged 再提交
- 建议养成 `git add <file>` 指定文件的习惯，少用 `git add -A` 以防不小心加入无关文件

### 坑 7：不要在 /loop 动态模式下进行 Git 推送等交互式操作
**现象**：如果处于 `/loop` 动态模式，推送操作可能因需要交互确认（如 GitHub 认证）而被阻塞或无法正常完成。

**原因**：`/loop` 模式限制了交互式输入，而 `git push` 可能需要凭据管理器弹窗。

**解决方案**：
- git push 这类操作在普通对话模式下执行，不要在 `/loop` 循环期间执行
- 如果必须自动化推送，确保已配置好 Git credential helper（如 `git config --global credential.helper manager-core`）

“PMU 是我项目里比较有代表性的一个板级差异点。参考设计原先不是按 GEC6818 实际供电结构来的，而我根据原理图确认这块板实际使用的是 AXP228 做电源管理，负责多路电压轨输出和上电控制。所以我把设备树和相关配置改成匹配 AXP228 的方式，目的是让系统的供电描述、上电控制和板级硬件一致。这个工作让我对系统移植里‘不仅是外设接口，连供电架构也属于板级适配’这件事理解更深。”

如果他再追问更细，你可以补一句：
“像 AXP228 这类 PMU 不只是供电芯片，它还参与开机、复位、状态管理，所以它在系统移植里是很关键的板级资源。”

你现在要改掉“我只是改了地址”的表达。

你现在的原话是“根据手册用 AI 写出了一整个 LCD 设备树框架并填写了数值”。这句话要改成下面这种表达：

六、触摸适配：继续强调“硬件描述与驱动匹配”​
触摸你已经有不错的基础，可以继续这样表达：

这样既保住了项目真实性，又给自己留了空间。
